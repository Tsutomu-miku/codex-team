# codexm list 展示设计说明

## 1. 背景

`codexm list` 当前已经能展示账号 quota、排序结果和 ETA，但默认表格里仍然存在几个问题：

- 默认列信息密度不均，部分列对决策帮助有限
- `AVAILABLE` 和旧的 `CURRENT SCORE` 语义强相关，默认视图存在重复表达
- 双窗口 reset 信息默认同时展开，表格较宽
- 高亮层级偏重，短时不可用和长时不可用没有明显区分

这份文档只固定 `list` 的展示设计，不讨论代码结构或实现步骤。

## 2. 目标

重新定义 `codexm list` 的展示设计，使默认视图更像“切号决策面板”：

- 默认视图优先回答“账号是谁、当前还剩多少、还能撑多久、下一次何时恢复”
- verbose 视图再展开 score 和 ETA 的解释性细节
- 默认表头命名简洁、一致、可扫描
- 颜色层级与风险层级一致
- 当前账号标记、不可用标记、warning 输出各自承担单一职责

## 3. 非目标

本次不涉及以下内容：

- 不改变 `list` 的排序逻辑
- 不改变 `auto-switch` 的排序逻辑
- 不改变 `list --json` 的数据结构
- 不改变 quota / ETA / score 的计算口径
- 不把实现细节、数据来源或内部推导公式放进默认表格

## 4. 信息架构

### 4.0 Summary 区

默认表格上方允许出现简短 summary 区，用于提供“账号池整体视角”。

summary 区的目标不是重复表内逐行信息，而是回答：

- 当前一共有多少可用账号
- 高风险账号有多少
- 各 plan 的分布如何
- 整个账号池合起来大概还剩多少标准化容量

summary 区不应变成第二张表，也不应展开实现细节或推导过程。

### 4.1 默认视图

默认表格固定为以下列顺序：

| 列 | 含义 |
| --- | --- |
| `NAME` | 账号名，同时承载当前账号标记 |
| `IDENTITY` | 掩码后的稳定身份标识 |
| `PLAN` | 账号 plan |
| `SCORE` | 当前综合可用度百分比 |
| `ETA` | 预计变为不可用的剩余时间 |
| `5H USED` | 当前 5 小时窗口使用率 |
| `1W USED` | 当前 1 周窗口使用率 |
| `NEXT RESET` | 当前账号下一次恢复可用性的关键时间点 |

默认视图不再保留以下列：

- `AVAILABLE`
- 旧的 `CURRENT SCORE`
- `REFRESH STATUS`
- `5H RESET AT`
- `1W RESET AT`

设计原因：

- `AVAILABLE` 与 `SCORE` 在默认视图里表达高度重叠
- 默认视图当前统一使用 `SCORE`
- `REFRESH STATUS` 在成功场景下大多恒为 `ok`，信息密度过低
- reset 默认只保留一个对决策最重要的时间点

### 4.3 默认 Summary 行

默认 summary 建议使用两行：

```text
Accounts: 5/7 usable | blocked: 1W 1, 5H 2 | plus x3, pro x2, team x2
Total: bottleneck 0.39 | 5H->1W 0.39 | 1W 0.42 (plus 1W)
```

其中：

- `Accounts` 行回答账号池结构和风险分布
- `Total` 行回答账号池的标准化剩余容量

不在默认 summary 中放入以下信息：

- 当前账号
- best candidate
- 平均 score
- 原始 5H / 1W 百分比总和

设计原因：

- 当前账号已在表前单独表达
- best candidate 已由表格排序自然表达
- 平均值对决策帮助有限
- 跨 plan 的原始百分比总和容易误导

### 4.2 Verbose 视图

`list --verbose` 在默认视图基础上展开解释性列，但仍保持分组清晰。

建议 verbose 视图按以下顺序追加：

1. ETA 解释组
   - `ETA 5H->1W`
   - `ETA 1W`
   - `RATE 1W UNITS`
   - `5H REMAIN->1W`
2. Score 解释组
   - `SCORE`
   - `1H SCORE`
   - `5H->1W 1H`
   - `1W 1H`
   - `5H:1W`
3. 双窗口恢复时间组
   - `5H RESET AT`
   - `1W RESET AT`

其中：

- verbose 仍允许出现双窗口 reset
- 当前 verbose 视图统一使用 `SCORE` / `1H SCORE`
- `1W:5H` 的旧推导列已移除，当前直接展示 `5H:1W`

## 5. 列设计

### 5.1 `NAME`

`NAME` 继续承担当前账号标记，不新增单独的 `CURRENT` 列。

显示规则：

- 当前账号：`* quota-main`
- 非当前账号：`  quota-backup`

设计原因：

- 当前账号是账号身份信息，不是健康状态
- 将其保留在 `NAME` 列可避免单独增加一列
- `*` 前缀对终端表格足够稳定，也便于快速扫描

### 5.2 `IDENTITY`

`IDENTITY` 保留掩码后的稳定标识，不做合并或弱化。

设计原因：

- 账号名不一定总是足够唯一
- 在团队或多快照场景下，`IDENTITY` 是定位具体账号的关键列

### 5.3 `PLAN`

表头从 `PLAN TYPE` 收敛为 `PLAN`。

设计原因：

- 信息不变
- 表头更短，与 `NAME`、`ETA`、`SCORE` 风格一致

### 5.4 `SCORE`

`SCORE` 直接展示当前综合 score 的百分比文本，例如：

- `87.5%`
- `18.0%`
- `0.0%`

默认视图不再显示 `available (62.5%)` 或 `available / 62.5%` 之类的双段式文本。

设计原因：

- 用户真正需要的是“当前还剩多少”
- `AVAILABLE` 与 `SCORE` 分拆后存在默认视图语义重复
- 百分比 + 颜色足以表达当前健康度

### 5.5 `ETA`

`ETA` 保持为当前预测值，例如：

- `1.8h`
- `25m`
- `-`

默认视图不展开 ETA 的来源解释。

设计原因：

- 默认视图只需要告诉用户“还能撑多久”
- 推导细节属于 verbose 信息

### 5.6 `5H USED` 和 `1W USED`

这两列继续保留，不合并为单列。

设计原因：

- 两个窗口代表不同约束
- 用户仍然需要快速判断“是短窗紧张还是长窗紧张”
- 它们与 `SCORE` 的角色不同：`SCORE` 是综合值，`5H USED / 1W USED` 是原因侧信号

### 5.7 `NEXT RESET`

`NEXT RESET` 只显示当前账号下一次恢复可用性的关键时间点。

显示形式：

- `MM-DD HH:mm`
- 若 reset 在短时间内发生，后缀显示倒计时，例如 `MM-DD HH:mm (5m)`

默认视图不同时展示两个窗口的 reset。

规则：

- 若账号当前已被 `1W` 窗口阻塞，则显示 `1W RESET AT`
- 若账号当前已被 `5H` 窗口阻塞且 `1W` 未阻塞，则显示 `5H RESET AT`
- 若 `5H` 和 `1W` 同时阻塞，则显示两者中更晚的那个 reset，因为那才是账号真正恢复可用的时间
- 若账号当前仍可用，则显示当前更关键约束窗口对应的下一次 reset

设计原因：

- 默认视图里同时放 `5H RESET AT` 和 `1W RESET AT` 会显著加宽表格
- 用户默认更关心“这个账号什么时候真正恢复可用”，而不是“最早哪个窗口先 reset”

## 6. 展示格式

### 6.0 Summary 文案格式

`Summary` 行使用固定信息顺序：

```text
Accounts: <x>/<n> usable | blocked: 1W <a>, 5H <b> | plus x<c>, pro x<d>, team x<e>
```

规则：

- `usable` 表示当前仍可进入候选集的账号数
- `blocked: 1W <a>, 5H <b>` 表示阻塞账号按主阻塞窗口拆分后的计数
- 其中 `1W` 表示 1 周窗口已阻塞的账号数
- 其中 `5H` 表示 5 小时窗口已阻塞但 1 周窗口未阻塞的账号数
- 末尾 plan 段表示按 plan 聚合后的账号数分布

如果某个 plan 的账号数为 0，则不出现在 `plans` 段里。

设计原因：

- `usable` 和 `blocked` 拆解是默认 summary 中最有决策价值的池子级信号
- plan 分布能快速帮助用户理解账号池结构

### 6.1 Total 行格式

`Total` 行使用固定格式：

```text
Total: bottleneck <x> | 5H->1W <y> | 1W <z> (plus 1W)
```

其中：

- `5H` 表示所有账号剩余 5H 空间统一换算后的标准化池
- `1W` 表示所有账号剩余 1W 空间统一换算后的标准化池
- `bottleneck` 取两者中更小的那个，表示当前账号池真正受限的总容量

这里的 `5H` 和 `1W` 是来源窗口标签，不是单位名。

默认输出在行尾保留统一单位说明。

设计原因：

- 用户先看总量拆分，再看单位说明，阅读顺序更自然
- 将来源窗口标签与统一归一化单位同时保留，能减少误解

### 6.1 表头命名

默认视图固定为：

```text
NAME | IDENTITY | PLAN | SCORE | ETA | 5H USED | 1W USED | NEXT RESET
```

表头命名要求：

- 尽量使用单词或短词组
- 同一张表内避免同时存在过长和过短、风格差异过大的表头
- 表头优先表达用户关心的业务语义，而不是内部字段名

### 6.2 单元格格式

格式规则如下：

- `SCORE` 使用百分比文本，不附加额外状态单词
- `ETA` 使用紧凑时间格式
- `USED` 使用百分比文本
- `NEXT RESET` 使用绝对时间，必要时附加短倒计时
- 当前账号仅通过 `NAME` 列前缀 `*` 表达

不使用以下格式：

- `available / 62.5%`
- `available (62.5%)`
- `* available / 62.5%`

设计原因：

- 避免在单个单元格里混入多类语义
- 保持默认表格的列对齐稳定

### 6.3 Summary 与 Total 的缺失值

当 summary / pool 由于数据不足无法完整计算时：

- `Summary` 里无法计算的计数项使用 `0` 仅在语义明确时显示
- `Total` 里无法计算的项使用 `-`

例如：

```text
Total: bottleneck - | 5H->1W - | 1W 0.42 (plus 1W)
```

不引入 `unknown pool`、`partial` 等额外文案。

设计原因：

- summary / total 仍应保持紧凑
- 默认输出不应引入第二套异常状态词汇

### 6.4 占位与缺失值

当 quota 数据不足以支持完整展示时，默认表格使用统一占位规则：

- 无法计算的 `SCORE`：`-`
- 无法计算的 `ETA`：`-`
- 缺失的 `USED` 值：`-`
- 缺失的 `NEXT RESET`：`-`

当账号 quota 处于 `unsupported`、`error` 或其他非正常刷新状态时：

- 默认表格仍保留该账号行
- 行内派生列优先使用 `-` 占位，不发明新的表格状态单词
- 具体错误原因继续放在表后 warning / failure 区域

设计原因：

- 默认表格保持紧凑，不把异常解释塞回主表
- 用统一占位比引入多种异常文案更容易扫描

### 6.5 Warning 和失败输出

默认表格之外的异常信息继续放在表后输出：

- quota refresh failures
- 账号异常 warning
- 其他补充提示

这些信息不回填到默认表格里作为单独列。

## 7. 标色设计

### 7.1 `SCORE` 列颜色

`SCORE` 承担默认视图里的主风险颜色。

建议规则：

- `0%`：红色加粗
- `0% < SCORE < 20%`：黄色加粗
- `20% <= SCORE < 80%`：默认色
- `80% <= SCORE < 100%`：绿色
- `100%`：绿色加粗

设计原因：

- `SCORE` 是默认视图里的综合健康指标
- 颜色应直接服务于“能否继续用、需不需要尽快切”
- 健康账号需要有正向识别，但不能让中段分值过度抢眼

### 7.2 `5H USED` / `1W USED` 颜色

这两列只保留高消耗预警，不承担正向健康提示。

建议规则：

- `< 80% used`：默认色
- `80% <= used < 100%`：黄色加粗
- `100%`：红色加粗

设计原因：

- 这两列是原因信号，不是主判断信号
- 若它们与 `SCORE` 使用同级强色，会造成视觉竞争

### 7.3 `NEXT RESET` 颜色

`NEXT RESET` 的绝对时间保持默认色；
仅当 reset 很快到来时，可对倒计时后缀做弱强调。

设计原因：

- reset 是辅助恢复信息，不应盖过 `SCORE`

### 7.4 整行高亮

整行红底只保留给 **`1W` 窗口不可用** 的账号。

不对所有 `unavailable` 账号都整行染红。

设计原因：

- `1W` 不可用通常意味着更长恢复周期，是更高等级的风险
- `5H` 不可用但 `1W` 仍可恢复时，整行红底过重
- 将“短时不可用”和“长时不可用”区分开来，能让默认表更符合实际决策优先级

### 7.5 无颜色降级

在不支持 ANSI 颜色、关闭颜色输出、或颜色不可读的终端环境里：

- 表格结构和列顺序保持不变
- `SCORE` 仍直接显示百分比文本
- 当前账号仍通过 `NAME` 前缀 `*` 标识
- 高风险账号不额外引入新的文本标记替代颜色

设计原因：

- 保持有色与无色输出的一致布局
- 避免为无色模式再定义第二套状态词汇

## 8. 风险层级

默认视图里的风险层级固定为：

1. `1W` 不可用：整行红底
2. `SCORE = 0%`：`SCORE` 红色加粗
3. `0% < SCORE < 20%`：`SCORE` 黄色加粗
4. `USED >= 100%`：对应 `USED` 红色加粗
5. `80% <= USED < 100%`：对应 `USED` 黄色加粗
6. `80% <= SCORE < 100%`：`SCORE` 绿色
7. `100%`：`SCORE` 绿色加粗
8. reset 即将到来：`NEXT RESET` 倒计时弱强调

这些层级不应互相混淆。

若同一账号同时满足多个条件，按更高层级展示，不叠加额外文本说明。

## 9. 一致性要求

### 9.1 当前账号标记

当前账号标记固定为 `NAME` 前缀 `*`，不在其他列重复表达。

### 9.2 状态语义

默认视图里：

- “当前是否可继续用”由 `SCORE` 的数值和颜色表达
- 不再重复输出单独的 `available` / `unavailable` 列

### 9.3 默认与 verbose 分层

默认视图只保留高价值决策信息；
verbose 视图负责回答“为什么这个分数是这样、两个窗口分别如何影响排序”。

### 9.4 `NEXT RESET` 选择原则

`NEXT RESET` 默认显示当前瓶颈约束对应的恢复时间。

如果两个窗口对当前约束同等关键，则显示更早的那个 reset。

设计原因：

- 默认视图只能容纳一个 reset 列
- 当存在并列瓶颈时，用户仍然更关心最先到来的恢复点

### 9.5 Total 的标准化口径

`Total` 行虽然显示为：

```text
Total: bottleneck <x> | 5H->1W <y> | 1W <z> (plus 1W)
```

但三者都必须使用同一标准化口径后再比较和求和。

默认设计中，内部统一基准继续使用 `plus` 的 `1W` 等价单位。

如果需要在设计文档或解释性文档中引用 plan 之间的容量比例，固定写为：

- `16:100`

设计原因：

- 该写法直接对应当前约定的标准化口径
- 设计文档应优先保持与实现口径一一对应，而不是再引入额外化简写法

## 10. 最终设计摘要

默认视图应回答四个问题：

1. 这是哪个账号
2. 当前综合上还剩多少
3. 预计还能撑多久
4. 下一次关键恢复点是什么时候

因此默认表固定为：

```text
NAME | IDENTITY | PLAN | SCORE | ETA | 5H USED | 1W USED | NEXT RESET
```

其中：

- `SCORE` 是默认视图主判断列
- `NAME` 前缀 `*` 表示当前账号
- 整行红底只用于 `1W` 不可用
- 其他异常信息放表后，不进入默认列
